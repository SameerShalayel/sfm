# Stage-Gate SDLC Methodology

## الفكرة الأساسية

من **عرض مشكلة مكون من 3-6 أسطر**، يتم بناء نظام برمجي كامل عبر 7 مراحل متسلسلة. كل مرحلة تنتج مخرجات محددة على شكل **nodes** (عقد بيانات بصيغة JSON)، وتنتهي ببوابة (**Gate**) تتطلب موافقة إنسان.

## المراحل السبع

```
المشكلة (3-6 أسطر)
    │
    ▼
┌─────────────────────────────────────────────────┐
│  PD — Problem Discovery (اكتشاف المشكلة)       │
│  ├── Problem Summary (ملخص المشكلة)             │
│  ├── Actors (الأطراف المتأثرة)                  │
│  ├── Goals (الأهداف)                             │
│  ├── Pain Points (نقاط الألم)                    │
│  ├── Constraints (القيود)                        │
│  ├── Scope (النطاق: in/out)                      │
│  ├── Success Signals (مؤشرات النجاح)             │
│  ├── Unknowns (أسئلة مفتوحة)                     │
│  └── 🚪 PD Lock Gate [Human Only]                │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S0 — Problem Lock (قفل المشكلة)                │
│  ├── Problem Lock (تأكيد تعريف المشكلة)         │
│  ├── Problem Insight (فهم جذر المشكلة)          │
│  ├── Desired Outcome (النتائج المرجوة)           │
│  ├── Strategic Direction (أين نروح وأين لا)     │
│  └── 🚪 Problem Gate [Human Only]                │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S1 — Product Shape (شكل المنتج)                │
│  ├── Product Shape (رؤية المنتج + MVP)           │
│  ├── Product Insight (فهم المنتج العميق)         │
│  ├── User Personas (المستخدمون وسلوكياتهم)       │
│  ├── Feature Modules (مجموعات الميزات)           │
│  ├── User Procedures (سير عمل المستخدم)          │
│  ├── MVP Boundary (حدود النسخة الأولى)           │
│  └── 🚪 Product Gate [Human]                     │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S2 — Architecture (الهيكل المعماري)             │
│  ├── Architecture Spine (التقنيات المختارة)      │
│  ├── Domain Model (المفاهيم الأساسية)            │
│  ├── System Components (المكونات المعمارية)       │
│  ├── Component Interactions (التواصل بين المكونات)│
│  ├── Data Domains (نموذج البيانات)               │
│  ├── Tech Constraints (القيود التقنية)           │
│  └── 🚪 Architecture Gate [Human]                │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S3 — Development (التخطيط للتنفيذ)              │
│  ├── Production Slice (الشريحة العمودية)          │
│  ├── Feature Slice (تفصيل الشريحة)               │
│  ├── API Contract (العقود البرمجية)               │
│  ├── Data Model (مخطط البيانات)                   │
│  ├── Task Breakdown (خطة المهام)                  │
│  └── 🚪 Slice Gate [Human + AI]                   │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S4 — Build (البناء الحقيقي)                     │
│  ├── Code Implementation (تنفيذ الكود)            │
│  ├── Code Structure (هيكل الملفات)                │
│  ├── Frontend/Backend Code (الكود الفعلي)         │
│  ├── Code Validation (مراجعة التتبع)              │
│  ├── Integration (تكامل المكونات)                 │
│  └── 🚪 Code Gate [Human + AI]                    │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S5 — Release (الاختبار والجودة)                 │
│  ├── Quality Assurance (ضمان الجودة)              │
│  ├── Test Plan (خطة الاختبار)                     │
│  ├── Unit Tests (اختبارات الوحدة)                 │
│  ├── Integration Tests (اختبارات التكامل)         │
│  ├── Acceptance Tests (اختبارات القبول)            │
│  ├── Bug Tracking (تتبع الأخطاء)                  │
│  └── 🚪 Quality Gate [Human]                      │
└─────────────────────────────────────────────────┘
    │
    ▼
┌─────────────────────────────────────────────────┐
│  S6 — Learn & Iterate (النشر والتعلم)            │
│  ├── Production Release (الإصدار)                 │
│  ├── Release Package (حزمة الإصدار)               │
│  ├── Deploy Checklist (قائمة النشر)                │
│  ├── Rollback Plan (خطة التراجع)                  │
│  ├── Documentation (التوثيق)                      │
│  ├── Handoff (التسليم)                             │
│  └── 🚪 Release Gate [Human Only]                 │
└─────────────────────────────────────────────────┘
```

## القواعد الأساسية

1. **لا يمكن تخطي مرحلة** — كل مرحلة تعتمد على مخرجات المرحلة السابقة
2. **البوابات إلزامية** — لا انتقال بدون موافقة إنسان
3. **التتبع (Traceability)** — كل مخرج يرجع لمرجع في المراحل السابقة
4. **المخرجات بصيغة JSON** — كل مرحلة تنتج ملف JSON يحتوي nodes + edges
5. **القيود أولاً** — حدد ما لا تبنيه قبل ما تبني

## أنواع البوابات

| البوابة | المرحلة | صلاحية القرار |
|---------|---------|---------------|
| PD Lock Gate | PD | Human Only |
| Problem Gate | S0 | Human Only |
| Product Gate | S1 | Human |
| Architecture Gate | S2 | Human |
| Slice Gate | S3 | Human + AI |
| Code Gate | S4 | Human + AI |
| Quality Gate | S5 | Human |
| Release Gate | S6 | Human Only |

## متى تستخدم هذه المنهجية

- عند بناء أي نظام برمجي جديد
- عند تلقي وصف مشكلة من 3-6 أسطر
- عندما يريد المستخدم منهجية منظمة ومتتبعة
- لضمان أن الحل يطابق المشكلة وليس العكس

## المخرجات المتوقعة

لكل مشروع، يتم إنتاج 7-8 ملفات JSON:
- `{project_id}_pd.json` — اكتشاف المشكلة
- `{project_id}_s0.json` — قفل المشكلة
- `{project_id}_s1.json` — شكل المنتج
- `{project_id}_s2.json` — الهيكل المعماري
- `{project_id}_s3.json` — التخطيط للتنفيذ
- `{project_id}_s4.json` — البناء
- `{project_id}_s5.json` — الاختبار
- `{project_id}_s6.json` — النشر والتسليم
