# Playbook — دليل التشغيل الاستراتيجي

## الغرض

هذا الـ Playbook يشرح **كيف تفكر** في كل مرحلة — القرارات الاستراتيجية، الفخاخ الشائعة، ومتى تتوقف.
الـ INSTRUCTION يشرح **ماذا تنتج**. هذا الـ Playbook يشرح **لماذا ومتى وكيف تقرر**.

---

## القاعدة الذهبية

> **لا تبني شيئاً قبل أن تفهم المشكلة.**
> إذا بدأت بالكود قبل PD/S0/S1/S2 — ستبني الشيء الخطأ.

## كيف تقرأ عرض المشكلة (3-6 أسطر)

### الخطوة 1: حدد الأطراف
اقرأ النص وابحث عن:
- **من** يتأثر (مرضى، موظفة، أطباء...)
- **من** يسبب المشكلة (النظام الحالي، الورقة، الإجراء...)
- **من** يملك القرار (المدير، صاحب العيادة...)

### الخطوة 2: حدد الألم
ابحث عن كلمات مثل:
- "لخبطة"، "غلط"، "ضياع"، "يزعلون"، "يخسر"
- هذه هي **نقاط الألم** — وثّقها كلها

### الخطوة 3: حدد القيود الضمنية
ما لم يُذكر في المشكلة عادةً يكون **خارج النطاق**:
- إذا المشكلة عن "حجوزات" — لا تبني نظام فواتير
- إذا المشكلة عن "3 أطباء" — لا تبني نظام 100 طبيب
- إذا المشكلة عن "عيادة صغيرة" — لا تبني enterprise system

### الخطوة 4: اسأل عن المجهول
حدد ما لم يذكره صاحب المشكلة:
- كم عدد [X] في اليوم؟
- هل [Y] يستخدم النظام مباشرة؟
- هل فيه حالات خاصة (إجازات، عطل)؟
- عربي فقط أو عربي/إنجليزي؟

## استراتيجيات كل مرحلة

### PD — اكتشاف المشكلة

#### الهدف
فهم المشكلة من كل الزوايا **قبل** التفكير بأي حل.

#### الفخاخ الشائعة
| الفخ | الأثر | الحل |
|------|-------|------|
| القفز للحل | تبني الشيء الخطأ | اكتب "ما لا نبنيه" قبل "ما نبنيه" |
| أهداف تقنية | المستخدم لا يهتم بالتقنية | اكتب أهداف من وجهة نظر المستخدم |
| نقاط ألم عامة | لا تساعد في التصميم | كل نقطة ألم تحتاج مثال محدد |
| نطاق مفتوح | المشروع لا ينتهي | فيه in وفيه out — كلاهما إلزامي |

#### متى تنتقل
- ✅ كل checklist في البوابة مستوفاة
- ✅ الإنسان وافق صراحةً
- ❌ لا تنتقل إذا فيه unknowns مفتوحة مهمة

---

### S0 — قفل المشكلة

#### الهدف
تكثيف PD إلى **4 نقاط واضحة**: المشكلة، الجذر، النتيجة، الاتجاه.

#### القرارات الاستراتيجية
1. **Problem Insight**: لا تكتب الأعراض — اكتب **الجذر**
   - ❌ "الموظفة تنسى الحجوزات" (عرَض)
   - ✅ "لا يوجد مصدر واحد للحقيقة" (جذر)

2. **Strategic Direction**: استخدم صيغة "نروح لـ / ما نروح لـ"
   - هذا يمنع scope creep في المراحل اللاحقة

#### متى تنتقل
- ✅ تقدر تشرح المشكلة لشخص خارجي بـ 30 ثانية
- ✅ الإنسان وافق

---

### S1 — شكل المنتج

#### الهدف
تحديد **ماذا** يفعل المنتج و**لمن** — بدون تفاصيل تقنية.

#### القرارات الاستراتيجية
1. **من المستخدم الأول؟** — إذا ما استخدمه هو، فشل المنتج
2. **البساطة vs الميزات** — في MVP دائماً البساطة تربح
3. **الألم الأكبر** — يجب أن يكون **مستحيل تقنياً** (مثل: Double Booking)

#### Product Insight: اسأل هذه الأسئلة
- إذا المستخدم الأول رفض يستخدم المنتج — لماذا؟
- ما الشيء الوحيد الذي إذا فشل، فشل المنتج كله؟
- هل البساطة أهم من الميزات لهذا المستخدم؟

#### MVP Boundary: القاعدة
```
إذا شككت — حطها في "خارج MVP"
أقل ميزات = أسرع تسليم = أسرع feedback = أفضل منتج
```

#### متى تنتقل
- ✅ User Personas واضحين بسلوكياتهم
- ✅ User Procedures مرقمة ومتسلسلة
- ✅ MVP Boundary صريح
- ✅ الإنسان وافق

---

### S2 — الهيكل المعماري

#### الهدف
ترجمة S1 إلى **قرارات تقنية** — ماذا نستخدم ولماذا.

#### القرارات الاستراتيجية

1. **اختيار التقنية**
   | حجم المشروع | التقنية المقترحة |
   |------------|-----------------|
   | صغير (1 ملف) | Vanilla HTML/CSS/JS + localStorage |
   | متوسط | Next.js/React + PostgreSQL |
   | كبير | Microservices + Docker + CI/CD |

2. **Domain Model**: كل كيان يجب أن يُشتق من S1 Feature Modules
   - إذا فيه feature module بدون كيان ← مشكلة
   - إذا فيه كيان بدون feature module ← dead entity

3. **Component Interactions**: كل سهم يجب أن يجيب على:
   - من المرسل؟
   - من المستقبل؟
   - ماذا يُرسل؟
   - ماذا يحدث عند الفشل؟

#### متى تنتقل
- ✅ Domain Model يغطي كل use cases من S1
- ✅ كل component له مسؤولية واحدة واضحة
- ✅ Data Model يدعم كل الـ constraints
- ✅ الإنسان وافق

---

### S3 — التخطيط للتنفيذ

#### الهدف
تحويل S2 إلى **خطة تنفيذ** مقسمة لشرائح عمودية (vertical slices).

#### القرارات الاستراتيجية

1. **أول شريحة**: اختر الشريحة التي تُثبت أن المنتج يعمل
   - مثال عيادة: "حجز موعد لطبيب واحد" (flow كامل end-to-end)
   - مثال مستشفى: "تسجيل مريض" (foundation للباقي)

2. **ترتيب الشرائح**:
   ```
   Foundation (الأساس) ← أولاً
   الشرائح المستقلة ← بالتوازي
   الشرائح المعتمدة ← أخيراً
   ```

3. **API Contracts**: حددها **قبل** كتابة أي كود
   - كل function: المدخلات + المخرجات + حالات الفشل
   - هذا يسمح بعمل متوازٍ بين frontend و backend

#### متى تنتقل
- ✅ كل شريحة تغطي flow كامل end-to-end
- ✅ API Contracts متوافقة مع S2
- ✅ Tasks متسلسلة ومرتبة
- ✅ Human + AI وافقوا

---

### S4 — البناء

#### الهدف
كتابة الكود الفعلي **بتتبع كامل** — كل سطر كود يرجع لمرجع في S1/S2/S3.

#### القرارات الاستراتيجية

1. **Budget (حد الأسطر)**: حدد حداً أقصى **قبل** ما تبدأ
   - مشروع صغير: 500 سطر
   - مشروع متوسط: 2000-5000 سطر
   - تجاوز الحد = مراجعة المعمارية

2. **Traceability (التتبع)**: كل function تحتاج:
   ```
   function bookSlot()    ← S3 Task 4
                           ← S2 BookingEngine component
                           ← S1 Feature: حجز موعد
                           ← PD Pain Point: Double Booking
   ```

3. **Code Validation قبل البوابة**:
   - لا dead code
   - localStorage/DB format يطابق S2 Data Model
   - كل endpoint يُستخدم من UI
   - Budget لم يُتجاوز

#### متى تنتقل
- ✅ كل task من S3 منفّذ
- ✅ Traceability كامل
- ✅ لا dead code
- ✅ Human + AI وافقوا

---

### S5 — الاختبار والجودة

#### الهدف
إثبات أن المنتج **يعمل بالضبط** كما حُدد في PD/S0/S1.

#### استراتيجية الاختبار

```
Level 1: Unit Tests ← اختبار Hard Rules من S1
Level 2: Integration Tests ← اختبار User Flows من S1
Level 3: Edge Cases ← اختبار حالات حدية
Level 4: Acceptance Tests ← اختبار Success Criteria من PD
Level 5 (مشاريع كبيرة): Security Tests + Cross-Slice Tests
```

#### كم اختبار؟
| حجم المشروع | Unit | Integration | Acceptance | إجمالي |
|------------|------|-------------|------------|--------|
| صغير | 5 | 7 | 5 | ~22 |
| متوسط | 15 | 15 | 10 | ~45 |
| كبير (4 شرائح) | 20 | 20 | 12 | ~85 |

#### ربط الاختبارات بالمراجع
كل اختبار يجب أن يحمل **معرّف مرجعي**:
- `T-HR-1` → Hard Rule 1 من S1
- `T-UF-3` → User Flow 3 من S1
- `T-SC-2` → Success Criteria 2 من PD

#### متى تنتقل
- ✅ 100% pass rate
- ✅ 0 أخطاء مفتوحة
- ✅ Quality Score ≥ 95
- ✅ الإنسان وافق

---

### S6 — النشر والتسليم

#### الهدف
نشر المنتج + تسليمه + توثيقه + التخطيط للتراجع.

#### Deploy Checklist (إلزامي)
قبل النشر، تأكد من:
- [ ] كل الاختبارات ناجحة
- [ ] Code review تم
- [ ] يعمل على كل المتصفحات المستهدفة
- [ ] Responsive يعمل على موبايل
- [ ] RTL/LTR حسب المطلوب
- [ ] البيانات تتحفظ بعد refresh
- [ ] لا console errors
- [ ] الأداء مقبول (< 1-3 ثوانٍ)

#### Rollback Plan (إلزامي)
```
ماذا لو فشل بعد النشر؟
├── السيناريو: [ما الذي سيفشل]
├── الخطة: [كيف نرجع]
├── الوقت: [كم يأخذ التراجع]
└── المسؤول: [من يتخذ القرار]
```

#### Handoff (التسليم)
```
لمن؟ → [أصحاب المصلحة]
ماذا يشمل؟ → [الملفات + الأدلة + التدريب]
دعم بعد التسليم؟ → [مدة المتابعة]
v2 → [ما القادم]
```

## أنماط القرار (Decision Patterns)

### النمط 1: "هل أضيف هذه الميزة؟"
```
هل هي في MVP Boundary (S1)؟
├── نعم ← ابنِها
├── لا ← لا تبنِها
└── غير واضح ← اسأل الإنسان عبر البوابة
```

### النمط 2: "أي تقنية أختار؟"
```
ما حجم المشكلة؟
├── صغير (1-3 شاشات) ← ملف HTML واحد + localStorage
├── متوسط (4-10 شاشات) ← Framework + Database
└── كبير (10+ شاشات) ← Full stack + Auth + CI/CD
```

### النمط 3: "كيف أقسم الشرائح؟"
```
أي شريحة تُثبت أن المنتج يعمل؟ ← Slice 1
أي شريحة تعتمد على Slice 1؟ ← بعد Slice 1
أي شريحة مستقلة؟ ← بالتوازي مع غيرها
```

### النمط 4: "هل الاختبار كافٍ؟"
```
كل Hard Rule مختبر؟ ← نعم ← ✅
كل User Flow مختبر end-to-end؟ ← نعم ← ✅
كل Success Criteria من PD متحقق؟ ← نعم ← ✅
أي شيء ← لا ← أضف اختبارات
```

### النمط 5: "متى أتوقف وأسأل الإنسان؟"
```
عند كل بوابة (Gate) ← إلزامي
عند unknowns مفتوحة تؤثر على القرار ← إلزامي
عند تجاوز Budget ← إلزامي
عند اكتشاف تعارض بين مرحلتين ← إلزامي
```

## مثال عملي مختصر

### المدخل:
```
محل قهوة صغير. الزبائن يستنون طويل والموظف ما يلحق يكتب الطلبات.
كل يوم ينكتب طلب غلط أو ينتسى.
```

### PD الناتج (ملخص):
- **Actors**: زبون، باريستا، صاحب المحل
- **Pain**: طلب غلط، نسيان طلبات، انتظار طويل
- **Goal**: طلب رقمي ← تأكيد فوري ← لا أخطاء
- **Scope IN**: تسجيل طلب + عرض الطلبات + تأكيد الاستلام
- **Scope OUT**: دفع إلكتروني، ولاء، مخزون
- **Constraint**: ملف واحد، بدون سيرفر، عربي

### S1 الناتج (ملخص):
- **MVP**: تسجيل طلب → إرساله → عرضه للباريستا → تأكيد الجاهزية
- **Persona 1**: الزبون — يختار ويطلب
- **Persona 2**: الباريستا — يشوف الطلبات ويؤكد
- **Hard Rule**: لا يمكن تعديل الطلب بعد الإرسال

### ... وهكذا لكل مرحلة حتى S6.
